# 差分保存の運用前提

この追補は、`upload_diff_design.md` の補足です。
差分保存の対象を **授業の教室配当のみ** に限定し、教室マスタは差分対象に含めない前提を明文化します。

## 前提

- 差分対象は **授業の教室配当のみ**
- **教室マスタは差分対象に含めない**
- コンフリクト時は **B を正** とする
- ただし、上書きした差分内容は **ポップアップで確認できる**

## 反映の考え方

1. B が編集した配当差分を、最新のクラウド状態に再適用する
2. A と B が同じ配当箇所を別々に変更していた場合でも、B の変更を採用する
3. そのときの差分内容は、書き込み前に確認できるようにする

## 差分の扱い

- 差分のキーは、授業側の配当を識別できる単位とする
- 追加・更新・削除は、いずれも配当差分として扱う
- 教室マスタの更新・追加・削除は、今回の差分保存では扱わない

## 注意点

- 差分反映は単純な全量上書きではない
- ただし、差分対象を配当に絞るので、マージ範囲は狭く保てる
- 教室マスタの変更は、別の書き込み経路で扱う

## 実装上の整理

- `書込` は、最新クラウドへ配当差分を反映する
- `取得` は、クラウドの最新配当をローカルへ反映する
- 差分確認モーダルでは、配当差分の内容を確認し、必要なら CSV で控えを残せるようにする

## 運用イメージ

- 同じログインIDを複数端末で使っていても、各端末のローカル変更を最新クラウドへ差分反映できる
- ただし、同じ配当箇所を複数端末が同時に変えた場合は、B 側の差分を優先する
- そのため、完全な排他制御ではなく、**最新差分を優先反映する運用** として使う
