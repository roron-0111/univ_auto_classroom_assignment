# 差分保存方式の設計

## 目的

現在のクラウド同期は、ローカルとクラウドのデータを **全量置換** する方式である。  
この方式では、同じキャンパスを複数人・複数端末で触っている場合に、最後に書き込んだ人の内容で上書きされやすい。

そこで、更新時に **全量上書きではなく差分保存** を行う方式へ切り替える。  
この設計の目的は次のとおり。

- ローカルの変更を必要な分だけクラウドへ反映する
- 既存のクラウドデータをできるだけ壊さない
- 同時編集時の上書き事故を減らす
- 「書込」と「取得」の意味をより自然にする

## 前提

- ログイン方式は現行のキャンパス別 Auth を維持する
- `hakkei@campus.local` のような共有アカウント運用は変えない
- ただし、**保存単位** は全量スナップショットではなく差分中心にする
- `取得` はクラウドの最新を反映する
- `書込` はローカル変更をクラウドへ反映する

## 基本方針

差分保存は、次の 2 層で考える。

1. **データ差分**
   - 授業
   - 教室
   - 配当
   - 配当ルール
   - 機材設定
   - taxonomoy / 候補マスタ

2. **同期差分**
   - 直前に読み込んだクラウド版と比べて、どの項目が変わったか
   - どの項目を追加・更新・削除するか

この 2 層を分けることで、単なる全量置換ではなく、  
**変更された部分だけを安全に反映する** ことを目指す。

## 何を差分保存するか

差分対象は、少なくとも次のデータ群とする。

- `subjects`
- `classrooms`
- `allocations`
- `settings`
- `equipmentSettings`
- `subjectTaxonomy`

### 差分単位の基本

- `subjects` は `subject.id` または `code` をキーに差分化する
- `classrooms` は `classroom.id` をキーに差分化する
- `allocations` は `subjectId + classroomId` の組で差分化する
- `settings` は項目単位で差分化する
- `equipmentSettings` と `subjectTaxonomy` はキャンパス単位マスタとして差分化する

## 差分保存の考え方

### 1. 追加
- ローカルに新しく増えたレコードをクラウドへ追加する

### 2. 更新
- 同じキーを持つ既存レコードを、変更された項目だけ上書きする

### 3. 削除
- ローカルから削除されたレコードを、クラウド側でも削除する

### 4. 参照整合
- 授業が削除されたら、その授業に紐づく配当も同時に削除する
- 教室が削除されたら、その教室を参照する配当も無効化する

## 重要な設計判断

差分保存を行うときに一番難しいのは、**どこまでを「差分」とみなすか** である。

### 推奨
- 授業や教室のような実体は「ID 単位」で差分を取る
- 配当は「授業ID + 教室ID」で差分を取る
- ルールや設定はキー単位で差分を取る

### 避けたいこと
- 1 件の大きな JSON を丸ごと比較して、部分更新だけを雑に当てる
- ID のないテキスト一致だけで更新対象を推測する
- 配当の複数行を無理に 1 件へ潰す

## ローカルとクラウドの役割

### ローカル
- 編集作業用の作業領域
- 未保存の変更を持つ
- 画面表示の即応性を保つ

### クラウド
- 共有の正本
- 書込時に差分を受け入れる
- 取得時に最新を返す

## 書込時の流れ

`ローカル → クラウドへ書き込み` は、次の流れで行う。

1. ローカル変更を検出する
2. クラウドの最新 revision を読む
3. ローカルが最後に読んだ revision と比較する
4. 差分を計算する
5. 差分をクラウドへ反映する
6. 反映後に新しい revision を発行する
7. ローカル側の同期基準 revision を更新する

### 書込の特徴
- 全量上書きではない
- 変更された項目だけ反映する
- ただし、競合がある場合は別途判定する

## 取得時の流れ

`クラウド → ローカルに取得` は、次の流れで行う。

1. クラウドの最新 revision を読む
2. 必要なら差分情報を表示する
3. ローカルの未保存変更があれば警告する
4. クラウドの差分をローカルへ反映する
5. ローカル表示を再構築する

### 取得の特徴
- 差分だけを取り込むのではなく、**ローカルの基準状態をクラウド版に寄せる**
- ただし、ローカル未保存分がある場合は上書き前に確認する

## 競合解決

差分保存では、競合ルールが重要になる。

### 競合の定義
- 同じレコードを、ローカルとクラウドの両方で別々に更新している状態
- もしくは、ローカルが読んだ後にクラウドが更新された状態

### 基本方針
- **同じキーの同時更新は、原則として検出する**
- 検出できた場合は、自動で上書きせず確認を出す
- 危険な場合は保存を止める

### 競合回避の優先順位
1. まずは差分を計算する
2. 次に revision を確認する
3. 競合があれば保存を止める
4. 必要なら手動で再取得させる

## 差分CSV

差分保存と合わせて、クラウド差分を CSV で出力できるようにする。

### CSV に出すもの
- 種別
- 操作（追加 / 変更 / 削除）
- 対象
- ローカル内容
- クラウド内容

### 表示方針
- `キー` のような内部IDは出さない
- 人間が見て分かる表記にする
  - 教室は教室名
  - 授業は `コード_科目名`
  - 配当は `コード_科目名 / 教室名`

### 目的
- 書込前のバックアップ
- 差分確認
- 共有前の目視確認

## 同一ユーザー・複数端末の考え方

差分保存方式では、同一ユーザーが複数端末で入っている場合でも、  
**どの端末のローカル差分をクラウドに反映するか** が重要になる。

### 推奨
- 各端末は独立したローカル差分を持つ
- 書込時にクラウド revision と比較する
- 差分が古い場合は保存を止める

### 補足
- これにより、同じ `hakkei@campus.local` でも、古い端末の状態で勝手に全上書きする事故を減らせる
- ただし、完全な同時編集ロックではない

## Firestore 上の保存イメージ

現行の分割保存構造を維持しつつ、各 chunk に差分更新をかける。

- `user_data_meta/{uid}`
  - revision
  - 最終更新時刻
  - 同期メタ情報

- `user_data_payload/{uid}/subjects/*`
  - subjects の chunk

- `user_data_payload/{uid}/classrooms/*`
  - classrooms の chunk

- `user_data_payload/{uid}/allocations/*`
  - allocations の chunk

- `user_data_payload/{uid}/config/main`
  - settings / equipmentSettings / subjectTaxonomy

## メリット

- 全量上書きより事故が少ない
- 変更箇所が分かりやすい
- 差分CSV と相性がよい
- 同一ユーザーの複数端末運用でも、古い状態をそのまま上書きしにくい

## デメリット

- 実装は全量保存より複雑
- 差分の正規化ルールが必要
- 競合検出を省くと、かえって混乱しやすい

## 実装順の推奨

1. まず差分のキーを定義する
2. 次に `subjects / classrooms / allocations` の差分抽出を作る
3. `settings / equipmentSettings / subjectTaxonomy` の項目差分を作る
4. 差分 CSV を出力する
5. revision を使って競合検出を入れる
6. 最後に書込 / 取得の UI を差分ベースへ寄せる

## 現時点での結論

この差分保存方式は **実装可能** である。  
ただし、全量置換よりも設計が重くなるため、以下を前提に進めるのが安全である。

- 差分の粒度を明確に決める
- 競合時は自動上書きしない
- CSV で差分を確認できるようにする
- 同一ユーザーの複数端末は、編集権を別々のローカル状態として扱う

この設計により、「書込はローカル→クラウドの一方向上書き」「取得はクラウド→ローカルの一方向上書き」という
現在の基本方針を保ちながら、より安全な差分同期へ移行できる。
