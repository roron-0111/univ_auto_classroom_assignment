# キャンパス単位の二重ログイン防止設計

## 目的

現在のキャンパス選択ログインは、内部的には `hakkei@campus.local` などの固定メールを使っている。  
そのため、同じキャンパスで複数端末から同時に入ると、同一 UID のまま別端末も書き込みできてしまう。

この設計書では、**ログイン方式そのものは変えず**、キャンパスごとに 1 人だけを編集可とするための
「セッション貸出 + 閲覧のみモード」を定義する。

## 基本方針

- Auth は現行の email/password 方式を維持する
- `hakkei / muronoki / kannnai` などのキャンパス別アカウントはそのまま使う
- ただし、**同じキャンパスの同時編集は 1 セッションのみ**に制限する
- 2 人目以降はログイン自体はできても、**閲覧のみ**にする
- `書込` は編集セッションを持つ 1 人だけが実行できる
- `取得` は閲覧のみユーザーでも実行可能

## 何を「二重ログイン」とみなすか

ここでいう二重ログインは、同一キャンパスに対して

- A の PC でログイン済み
- B の PC でも同じキャンパスでログイン済み
- どちらも編集できる

という状態を指す。

この状態を避けるため、**キャンパスごとに 1 つの編集セッションしか保持しない**。

## 設計の考え方

### 1. ログインと編集権は分ける

Firebase Auth の成功は「その人がキャンパスに入れた」ことを意味するだけで、
「編集権を持つ」こととは分ける。

そのため、ログイン直後に Firestore 上のセッション情報を確認し、

- 空いていれば編集セッションを確保
- すでに他端末が使っていれば閲覧のみへ切り替え

という流れにする。

### 2. セッションは端末側の識別子で区別する

同じ Auth UID でも、端末やブラウザごとに別セッションとして扱う。

各端末は、初回ログイン時に sessionId を生成して保持する。
- 可能なら `localStorage` で保持する
- 同じブラウザプロファイルでは同じ sessionId を使う
- 別 PC / 別ブラウザでは別 sessionId になる

これにより、同じ `hakkei@campus.local` でも、端末ごとに「どちらが編集者か」を判定できる。

## Firestore 上の保持内容

現行の `user_data_meta/{uid}` を、編集セッションの管理にも使う。

保存する情報のイメージ:

- `activeSessionId`
- `activeSessionOwnerEmail`
- `activeSessionStartedAt`
- `activeSessionHeartbeatAt`
- `activeSessionExpiresAt`
- `activeSessionMode`
  - `edit`
  - `readOnly`

必要なら将来的に、書き込みロック用の短期情報とは別にするが、まずは同じメタドキュメントで扱う。

## ログイン時の流れ

1. ユーザーがキャンパスを選んでログインする
2. Auth が成功する
3. アプリは sessionId を生成または再利用する
4. Firestore の `user_data_meta/{uid}` を読む
5. 以下を判定する
   - `activeSessionExpiresAt` が切れている
   - `activeSessionId` が自分の sessionId と一致する
   - `activeSessionId` が空
6. 空いていれば自分を編集セッションとして登録する
7. すでに他セッションが有効なら、**閲覧のみモード**へ切り替える

## 閲覧のみモード

閲覧のみモードでは次の挙動にする。

- 一覧閲覧は可能
- `取得` は可能
- `書込` は不可
- `更新(共有)` のような旧概念は使わない
- 書き込み系のボタンは無効化または押下時に拒否
- 画面上に `現在、閲覧のみ可能です` を表示

## 編集セッションの維持

編集セッションは永久保持せず、**短い有効期限付きの lease** とする。

### 推奨値
- heartbeat: 30 秒前後
- lease 期限: 2〜3 分前後

### 更新方法
- 編集者は定期的に heartbeat を送る
- heartbeat のたびに `activeSessionHeartbeatAt` と `activeSessionExpiresAt` を更新する
- 画面を閉じた / フリーズした / 落ちた場合でも、期限を過ぎれば自然失効する

## ログアウト時の流れ

1. ユーザーが明示的にログアウトする
2. 自分の sessionId と一致する場合のみ、セッション情報を解除する
3. 解除に失敗しても、TTL によって自然失効する

## ブラウザを閉じた場合

`beforeunload` だけで完全に解除するのは保証しない。  
そのため、ブラウザを閉じても最終的には TTL で回収する。

必要なら補助として、

- 閉じる前にログアウト確認
- タブを閉じるときだけ軽い確認

を付けられるが、**本命は TTL** とする。

## 書込との関係

書込はこの編集セッションを持つ 1 人だけが行う。

### 書込時の判定
- 自分が編集セッション保持者か
- セッションが有効か
- 直前に他端末へ編集権が移っていないか

満たさない場合は、書込を止めて `現在、閲覧のみ可能です` を返す。

## 取得との関係

`取得` は閲覧のみでも許可する。

- セッション保持者でなくても可能
- 既存のローカルをクラウドで上書きする操作なので、必要なら確認モーダルを出す
- ただしロック取得は不要

## この方式の利点

- Auth の仕組みを変えずに済む
- 端末ごとの二重編集を抑止できる
- ブラウザを閉じたときの残留ロック問題を TTL で回収できる
- ローカル→クラウドの書込だけを守ればよいので、ロック設計が簡潔

## 注意点

- 同じキャンパスの編集者が 1 人に制限されるため、複数人の同時編集はできない
- ただし読み取りは複数端末で可能
- 「誰が編集者か」を UID ではなく sessionId で判定することが重要
- 現在の `hakkei@campus.local` のような共有アカウント運用は維持できるが、編集権は 1 セッションに限定される

## 実装前提の確認事項

- sessionId をどこに保持するか
  - 推奨: `localStorage`
- TTL をどこで管理するか
  - 推奨: `user_data_meta/{uid}`
- 閲覧のみモード時の UI 表示
  - `現在、閲覧のみ可能です`
- 書込ボタンの挙動
  - 押下時に拒否

## 実装順の目安

1. sessionId の生成と保持
2. Firestore のメタ文書への activeSession 保存
3. ログイン時の編集権判定
4. 閲覧のみモードへの切り替え
5. heartbeat と TTL
6. ログアウト解除
7. 書込ボタンの拒否処理

## 位置づけ

この設計は、現行の「書込時だけ短期ロック」より上位の安全策であり、  
**同一キャンパスの二重ログイン防止**を目的とする。

今の Auth 方式を維持したまま、最小の概念追加で安全性を上げるための設計メモである。
