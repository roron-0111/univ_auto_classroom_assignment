# キャンパス分離 設計メモ

このメモは、`八景 / 関内 / 室の木` のようにキャンパスを追加するときの参照元です。  
方針は **「画面と配当ロジックは極力そのまま、保存と読み込みの単位だけをキャンパスごとに分ける」** です。

## 1. 目的

- 現在ログイン中のキャンパスだけを閲覧・編集・削除できるようにする
- 教室マスタ、授業科目マスタ、配当結果、設定をキャンパス別に分離する
- 将来キャンパスを追加しても、既存キャンパスに干渉しないようにする

## 2. 非対象

- 教室自動配当アルゴリズム本体の変更
- 配当条件の再設計
- 画面レイアウトの全面変更
- キャンパスをユーザが自由に切り替える管理画面の追加

## 3. キャンパスの扱い

- ログイン中のメールアドレスから `currentCampusLabel` を決める
- 保存は `campusId` / `campusLabel` をキーにスコープ化する
- 例: `campus:八景:subjects`
- 現在のキャンパス以外のデータは読み込まない
- 追加後のキャンパスも同じ仕組みで独立スコープとして扱う

## 4. データの分離単位

キャンパスごとに分ける対象:

- 教室マスタ
- 授業科目マスタ
- 配当結果
- 配当設定
- 機材設定
- 表示設定

共通のままでよいもの:

- ログイン画面
- 配当ロジック本体
- 各一覧画面の基本レイアウト
- CSV の基本UI

## 5. 教室管理の方針

### 5-1. 教室データ

- 教室レコードに `campus` を明示的に持たせる
- `金沢八景` のような表記は内部で `八景` に正規化する
- 新規追加時は現在キャンパスを自動付与する

### 5-2. 画面表示

- 一覧には `キャンパス` 列を表示してよい
- ただし編集フォームで別キャンパスを選ぶプルダウンは置かない
- 編集時の `campus` は現在キャンパスに固定する
- つまり、`八景` ログイン中に `関内` を選んで保存することはできない

### 5-3. CSV 入出力

- CSV の列順に `キャンパス` を含める
- `キャンパス` は必須列とする
- インポート時は、ファイル内の `campus` が現在キャンパスと完全一致していることを確認する
- 1件でも異なる、または判定できないレコードがあれば全体を拒否する
- エラーメッセージは `キャンパスが異なるレコードがあります` に統一する
- 例外的な許可や自動補完はしない

## 6. 授業管理の方針

### 6-1. 授業データ

- 授業レコードにも `campus` を明示的に持たせる
- 新規追加・編集保存時は現在キャンパスを自動付与する
- CSV 読み込み時も現在キャンパスに揃える

### 6-2. 画面表示

- 一覧には `キャンパス` 列を表示してよい
- ただし編集フォームで別キャンパスを選ぶプルダウンは置かない
- `campus` は入力欄ではなく、読み取り専用または内部固定値として扱う

### 6-3. CSV 入出力

- CSV の列順に `キャンパス` を含める
- `キャンパス` は必須列とする
- インポート時は、ファイル内の `campus` が現在キャンパスと完全一致していることを確認する
- 1件でも異なる、または判定できないレコードがあれば全体を拒否する
- エラーメッセージは `キャンパスが異なるレコードがあります` に統一する
- 例外的な許可や自動補完はしない

## 7. 起動時の読み込み

- まず現在キャンパスの scoped key を読む
- scoped key がなければ、旧キーが残っている場合は読み替える
- 教室・授業ともに、読み込んだデータは現在キャンパスへ正規化する
- 旧 seed データは、表示上の見た目だけを整える

## 8. 自動配当との関係

- 配当アルゴリズム本体は触らない
- その時点で選択されているキャンパスの教室・授業だけを solver に渡す
- current campus と異なるデータが混ざっていれば、配当入口で止める
- 例外確認・再配置も同じキャンパス内のみで完結させる

## 9. 将来キャンパスを追加するときの考え方

- 追加はコードベースで行う
- `CAMPUSES` に追加して email 解決を寄せる
- 既存の `八景` データと新キャンパスのデータは相互に干渉させない
- Web 上の設定だけで完結させる前提にはしない

## 10. 実装時に最初に見るファイル

- `src/App.tsx`
- `src/types.ts`
- `src/components/ClassroomManager.tsx`
- `src/components/SubjectManager.tsx`
- `src/components/CloudConnectionModal.tsx`
- `src/utils/csvParser.ts`
- `src/data/mockData.ts`

## 11. 実装チェックリスト

- 現在キャンパス以外のデータが保存・読み込みされない
- 教室編集フォームにキャンパス切替の UI がない
- 授業編集フォームにキャンパス切替の UI がない
- CSV インポートが現在キャンパス不一致で必ず落ちる
- CSV エラーメッセージが固定文言になっている
- 自動配当・例外確認・再配置が current campus 以外を参照しない
- 新しいキャンパスを足しても既存キャンパスに影響しない

## 12. 最小実装手順

将来 `関内` や `室の木` を足すときは、次の順で進める。

1. `src/types.ts` の `CAMPUSES` にキャンパスを追加する
2. ログイン email とキャンパスの対応を追加する
3. `src/App.tsx` の `currentCampusLabel` から scoped key を読む流れをそのまま使う
4. `src/components/ClassroomManager.tsx` と `src/components/SubjectManager.tsx` で、編集フォームのキャンパスを現在値に固定する
5. CSV import/export の列順を現行キャンパスの仕様に合わせる
6. 現在キャンパスと異なる CSV は `キャンパスが異なるレコードがあります` で全体拒否する
7. `src/data/mockData.ts` の seed は、必要なら新キャンパス用にだけ調整する
8. 最後に `tsc -b --pretty false` で型を確認する

この順で進めれば、既存の `八景` 運用に干渉しにくく、将来の追加も局所的に済ませやすい。

## 13. 開講学部・管轄のキャンパス別マスタ

`開講学部` と `管轄` は、キャンパスごとに内容が異なる前提で扱う。

### 13-1. 目的

- `八景` の候補一覧と `関内` / `室の木` の候補一覧を分ける
- 候補の追加・編集・削除を、現在キャンパスごとに行えるようにする
- 授業データ本体の保存値は文字列のまま維持し、候補一覧だけをキャンパス別にする

### 13-2. データの持ち方

- 授業レコードの `faculty` / `department` は今までどおり文字列で持つ
- 候補一覧だけをキャンパス別に保存する
- 例: `campus:八景:facultyOptions` / `campus:八景:departmentOptions`
- 既定値は `CAMPUSES` の追加と同じように、キャンパスごとの seed から補完する

### 13-3. 画面の考え方

- 授業管理の編集画面では、現在キャンパスの候補一覧だけを表示する
- 別キャンパスの候補は見せない
- 一覧のフィルタや CSV 取り込みも、現在キャンパスの候補に合わせて判定する
- 候補マスタの編集 UI は、設定画面または管理画面として別枠に置く

### 13-4. 実装時の注意

- `src/types.ts` の固定配列に直接書くと、全キャンパス共通になってしまう
- `SubjectManager` と `SubjectEditModal` は、固定配列ではなくキャンパス別候補を読むようにする
- CSV インポート時は、候補外の `faculty` / `department` を弾くか、少なくとも現在キャンパスの候補に正規化する
- ただし、この変更は自動配当ロジックには触れない

### 13-5. 最小の実装スコープ

まずは次の順で進めると安全。

1. キャンパス別の候補一覧を読む helper を作る
2. `SubjectManager` のフィルタと編集欄を helper 参照に変える
3. `SubjectEditModal` のプルダウンを helper 参照に変える
4. CSV インポートで現在キャンパスの候補外値をチェックする
5. 必要なら候補編集 UI を追加する

これなら、授業データ本体と自動配当ロジックを壊さずに、キャンパスごとの差分だけを管理できる。

## 14. この機能で最初に見るファイル

`開講学部` と `管轄` をキャンパス別マスタに寄せるときは、まず次だけ見ればよい。

- `src/App.tsx`
- `src/components/SubjectManager.tsx`
- `src/components/SubjectEditModal.tsx`
- `src/utils/csvParser.ts`
- `src/data/mockData.ts`
- `src/types.ts`

実装時の基本方針は次のとおり。

- `src/types.ts` に候補マスタの共通型や helper を寄せる
- `SubjectManager` と `SubjectEditModal` は、その helper を読むだけにする
- `csvParser.ts` は現在キャンパス候補に合わない値を弾く
- `mockData.ts` は seed の候補を現在キャンパスに合わせる
- `App.tsx` は現在キャンパスの候補セットを渡すだけにする
